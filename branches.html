üß† Understanding HEAD in Git

The HEAD is a special pointer in Git. It always points to the latest commit on your current branch.

To see where HEAD is pointing, you can use:

This will show you recent commits and highlight where HEAD and your branch names are pointing.

The default branch used to be called master, but modern Git uses main. There‚Äôs nothing technically special about main ‚Äî it‚Äôs just a naming convention.

üîÄ Fast-Forward Merge in Git

A fast-forward merge is the simplest type of merge in Git. It happens when the branch you are merging is directly ahead of the current branch ‚Äî meaning there's no divergence or conflicts. Let's walk through a real example.

üìÅ Step-by-Step: Building a Navbar

Imagine you‚Äôre working on a basic website. You start by creating an index.html on the main branch. Then, you decide to build the navbar separately in a feature branch.

Step 1: Initialize your project and commit the base file

git add .
git commit -m "Initial commit with index.html"

Step 2: Create a new branch for the navbar feature

You don't want to mess up your main branch while experimenting, so you create a new one:

Step 3: Work on the navbar and commit the changes

Add a navbar in your index.html and commit it on the navbar branch.

git add index.html
git commit -m "Added navbar to index.html"

Step 4: Merge the feature back into main (Fast-Forward)

Before you run the merge command, make sure you are on the branch you want to merge into ‚Äî in this case, main.

Now, switch to the main branch and merge:

git checkout main
git merge navbar

Since no changes were made on main after the branch was created, this is a fast-forward merge ‚Äî Git simply moves the pointer forward.

Step 5: (Optional) Clean up your branch

If you‚Äôre done with the feature branch, you can delete it:

‚úÖ That‚Äôs it! You‚Äôve successfully merged a feature branch using fast-forward. Clean, simple, and no conflicts!

üîÄ 3-Way Merge (Non Fast-Forward)

A 3-way merge occurs when both branches have progressed independently after their common ancestor. Git uses the base (common ancestor), the source branch, and the target branch to combine changes. This is common in collaborative workflows.

Visual Representation: Blue = master commits, Red = feature commits

üìÅ Example Scenario

  • You're on main branch and create a new branch called feature.
  • On feature, you create a new file feature.html and commit your changes.
  • Meanwhile, your teammate commits changes to index.html on main.
  • Now both branches have unique new commits ‚Äî they have diverged.

üß† Important Note

You must be on the branch that you want the changes to be merged into.
For example: If you want to merge feature into main, first switch to main.

üîß Merge Steps

  1. Switch to the target branch (main):
    git checkout main
  2. Merge the feature branch:
    git merge feature
  3. Git will detect the different commit histories and perform a 3-way merge.
    If there are no conflicts, the merge completes automatically.
    If conflicts exist, Git marks them in files and asks you to resolve manually.
  4. Resolve the conflicts in a code editor, then:
    git add .
    git commit

Now main includes both the index.html updates and the new feature.html file ‚Äî a successful 3-way merge!
